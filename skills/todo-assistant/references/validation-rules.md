# Validation Rules

Generation 직후 1회 실행. 실패 항목은 최대 2회 재생성 후에도 실패하면 해당 항목을 사용자에게 질문.

## 검사 순서

1. **추상성 검사** — 모호 동사 blocklist 매칭
2. **근거 무결성** — sources 필드 검증
3. **개수·시간·난이도** — 일일 한도 검사
4. 모두 통과 → 사용자 컨펌 단계로 진행

## 1. 추상성 검사

**Blocklist (모호 동사):**
```
리뷰하기, 정리하기, 살펴보기, 공부하기, 확인하기,
검토하기, 파악하기, 알아보기, 정리, 학습
```

**필수 필드:**
- `target`: 작업 대상 (파일·모듈·문서·URL)
- `done_when`: 완료 조건 (관찰 가능한 기준)

**실패 조건:**
- title에 blocklist 동사만 포함되고 구체 명사 없음
- `target`이 비어있거나 일반적 단어("코드", "문서") 만
- `done_when`이 "다 했을 때" 같은 자기 참조

**재생성 프롬프트:** "다음 todo를 더 구체적으로 다시 써라. target에 실제 파일/모듈명, done_when에 관찰 가능한 산출물(파일·테스트 결과·수치)을 넣어라."

## 2. 근거 무결성

**필수 필드:** `sources: [{doc_id, locator}]`

**검사:**
- 각 source의 `doc_id`가 입력 자료 목록에 실재하는가
- `locator`(예: `L42-50`, `section: "Auth"`, `URL#anchor`)가 유효 범위인가

**실패 시:** 해당 todo를 재생성하면서 입력 자료 목록을 다시 제공.

## 3. 개수·시간·난이도

| 항목 | 한도 |
|------|------|
| 일일 todo 개수 | ≤ 5 |
| 일일 총 예상 시간 | ≤ 6h (360분) |
| hard 난이도 todo | ≤ 2 |
| 단일 todo 예상 시간 | ≤ 90분 |

**초과 시 자동 처리:**
- 우선순위 낮은 todo부터 weekly로 이동
- 시간이 90분 초과인 단일 todo는 분해
- 자동 처리 후에도 한도 초과면 사용자에게 분산 제안 ("다음 날로 미루기 / 주간으로 승격")

## 4. Gate 위치 정리

| 단계 | 위치 | 차단 조건 |
|------|------|-----------|
| Memory Compaction | 진입 직후 | 7일↑ or active≥30 |
| Status Check | Memory Load 직후 | 미완료 항목에 응답 없음 |
| Ambiguity | Analysis 직후 | unknowns 비어있지 않음 |
| Validation | Generation 직후 | 위 1~3 검사 실패 |
| User Confirm | Validation 통과 후 | 사용자 승인 없음 |

각 Gate는 통과하기 전에 다음 단계로 진입하지 않는다.
